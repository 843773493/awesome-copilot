# 产品需求文档（PRD）

## 概述

设计全面且符合生产标准的产品需求文档（PRD），弥合商业愿景与技术执行之间的差距。此技能适用于现代软件系统，确保需求被清晰定义。

## 何时使用

在以下情况下使用此技能：

- 启动新产品或功能开发周期
- 将模糊的想法转化为具体的规格说明
- 定义AI驱动功能的需求
- 利益相关者需要一个统一的“项目范围真相来源”
- 用户要求“撰写PRD”、“记录需求”或“规划功能”

---

## 操作流程

### 第一阶段：发现（访谈）

在撰写PRD的任何一行之前，**必须**通过深入访谈用户来填补知识空白。不要假设上下文。

**询问以下内容：**

- **核心问题**：为什么现在要开发这个功能？
- **成功指标**：如何知道它已经成功？
- **限制条件**：预算、技术栈或截止日期？

### 第二阶段：分析与范围定义

综合用户的输入，识别依赖项和隐藏的复杂性。

- 绘制**用户流程**。
- 明确**非目标**以保护项目时间线。

### 第三阶段：技术文档撰写

使用以下**严格PRD结构**生成文档。

---

## PRD质量标准

### 需求质量

使用具体且可衡量的标准。避免使用“快速”、“简单”或“直观”等模糊表述。

```diff
# 模糊（差）
- 搜索功能应快速且返回相关结果。
- 用户界面必须看起来现代且易于使用。

# 具体（好）
+ 搜索功能必须在10,000条记录数据集中于200毫秒内返回结果。
+ 搜索算法必须在基准测试中达到≥85%的Precision@10。
+ 用户界面必须遵循'Vercel/Next.js'设计系统，并实现100%的Lighthouse可访问性评分。
```

---

## 严格PRD结构

您**必须**按照以下精确结构输出文档：

### 1. 执行摘要

- **问题陈述**：1-2句话描述痛点。
- **解决方案**：1-2句话说明解决方式。
- **成功标准**：3-5个可衡量的KPI。

### 2. 用户体验与功能

- **用户画像**：该功能面向哪些用户？
- **用户故事**：`作为[用户]，我希望[操作]，以便[好处]。`
- **验收标准**：每个故事的“完成”定义列表。
- **非目标**：我们不打算开发哪些内容？

### 3. AI系统需求（如适用）

- **工具需求**：需要哪些工具和API？
- **评估策略**：如何衡量输出质量和准确性。

### 4. 技术规格

- **架构概述**：数据流和组件交互。
- **集成点**：API、数据库和认证。
- **安全与隐私**：数据处理和合规性。

### 5. 风险与路线图

- **分阶段发布**：MVP -> v1.1 -> v2.0。
- **技术风险**：延迟、成本或依赖项失败。

## 实施指南

### 应做（始终）

- **定义测试**：对于AI系统，明确如何测试和验证输出质量。
- **迭代**：提供初稿并要求对特定部分进行反馈。

### 不应做（避免）

- **跳过发现阶段**：绝不要在未至少询问两个澄清问题的情况下撰写PRD。
- **虚构限制条件**：如果用户未指定技术栈，请询问或标记为`待定（TBD）`。

## 示例：智能搜索系统

### 1. 执行摘要

**问题**：用户难以在大型仓库中找到特定文档片段。
**解决方案**：提供直接答案并附带来源引用的智能搜索系统。
**成功标准**：

- 将搜索时间减少50%。
- 引用准确性≥95%。

### 2. 用户故事

- **故事**：作为开发者，我希望用自然语言提问，而无需猜测关键词。
- **验收标准**：
  - 支持多轮澄清。
  - 返回带有“复制”按钮的代码块。

### 3. AI系统架构

- **所需工具**：`codesearch`、`grep`、`webfetch`。

### 4. 评估

- **基准测试**：使用50个常见开发者问题进行测试。
- **通过率**：必须与预期引用匹配90%。
