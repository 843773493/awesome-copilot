# 产品经理助手：功能识别与规格说明

此工作流程将引导您通过一种系统的方法来识别缺失的功能，进行优先级排序，并创建详细的实现规格说明。

## 1. 项目理解阶段

- 审查项目结构以了解其组织方式
- 阅读 README.md 和其他文档文件以了解项目的核心功能
- 通过检查以下内容来识别现有实现状态：
  - 主入口点（CLI、API、UI 等）
  - 核心模块及其功能
  - 测试以了解预期行为
  - 任何占位符实现

**指导性问题：**
- 该项目的主要目的是什么？
- 它解决了哪些用户问题？
- 当前实现中存在哪些模式？
- 文档中提到但尚未完全实现的功能有哪些？

## 2. 差距分析阶段

- 仅将文档中记录的功能与实际实现进行对比
- 识别缺乏实际功能的“占位符”代码
- 查找文档中提及但缺乏稳健实现的功能
- 考虑用户旅程并识别断裂或缺失的步骤
- 首先关注核心功能（而非锦上添花的功能）

**输出创建：**
- 创建一个潜在缺失功能的列表（5-7 项）
- 对每个功能，记录：
  - 当前实现状态
  - 文档中的引用
  - 如果缺失，对用户体验的影响

## 3. 优先级排序阶段

- 对每个识别到的差距进行评分：

**评分矩阵（1-5 分制）：**
- 用户影响：有多少用户受益？
- 战略对齐：是否符合核心使命？
- 实现可行性：技术复杂度？
- 资源需求：所需开发工作量？
- 风险等级：潜在的负面影响？

**优先级 = (用户影响 × 战略对齐) / (实现工作量 × 风险等级)**

**输出创建：**
- 根据评分展示优先级最高的前 3 个缺失功能
- 对每个功能，提供：
  - 功能名称
  - 当前状态
  - 未实现时的影响
  - 对其他功能的依赖关系

## 4. 规格说明开发阶段

- 对每个优先级排序的功能，开发详细但实用的规格说明：
  - 从哲学方法开始：以简单性优先于复杂性
  - 首先聚焦于最小可行产品（MVP）功能
  - 考虑开发者的体验
  - 保持规格说明便于实现

**每个功能规格说明：**
1. **概述与范围**
   - 它解决了什么问题？
   - 包含哪些内容？哪些内容明确排除？

2. **技术需求**
   - 所需的核心功能
   - 用户界面（API、UI、CLI 等）
   - 与现有代码的集成点

3. **实现计划**
   - 需要创建或修改的关键模块/文件
   - 展示方法的简单代码示例
   - 明确的数据结构和接口

4. **验收标准**
   - 如何知道它已经完成？
   - 必须正常工作的具体功能？
   - 应该通过的测试？

## 5. GitHub 问题创建阶段

- 对每个规格说明，创建一个 GitHub 问题：
  - 清晰、描述性的标题
  - 问题正文中的完整规格说明
  - 适当的标签（如增强功能、高优先级等）
  - 在相关部分明确提及最小可行产品（MVP）哲学

**问题模板结构：**

# [功能名称]

## 概述
[功能和其目的的简要描述]

## 范围
[包含的内容和明确排除的内容]

## 技术需求
[具体的技术需求和限制条件]

## 实现计划
[分步骤的方法，包含简单的代码示例]

## 验收标准
[明确的功能完成要求列表]

## 优先级
[优先级排序的依据]

## 依赖关系
- **被阻塞:** [列出此问题所阻塞的问题]
- **阻塞:** [列出此问题所依赖的问题]

## 实现规模
- **预估工作量:** [小/中/大]
- **子问题:** [如果这是父问题，列出子问题链接]

## 5.5 工作分配优化

- **独立性分析**
  - 审查每个规格说明以识别真正独立的组件
  - 重构规格说明以最大化独立工作流
  - 在相互依赖的组件之间建立清晰的边界

- **依赖关系映射**
  - 对于无法避免依赖的功能，建立清晰的问题层级关系
  - 为整体功能创建父问题，并为子组件创建子问题
  - 明确记录“被阻塞”和“阻塞”关系

- **工作量平衡**
  - 将大型规格说明分解为更小、更易管理的子问题
  - 确保每个子问题代表 1-3 天的开发工作量
  - 包含子问题特定的验收标准

**实现指南：**
- 使用 GitHub 问题链接语法创建显式的关联关系
- 添加标签以指示依赖状态（例如：“被阻塞”、“前提条件”）
- 为每个问题添加预估复杂度/工作量，以辅助冲刺计划

## 6. 最终审查阶段

- 总结所有创建的规格说明
- 突出功能之间的实现依赖关系
- 建议一个逻辑的实现顺序
- 记录任何潜在挑战或注意事项

在整个过程中，请记住：
- 优先简单性而非复杂性
- 从最小可行产品实现开始，确保其能正常工作
- 聚焦于开发者的体验
- 构建一个可后期扩展的基础
- 考虑开源社区和贡献模式

此工作流程体现了我们的方法，有助于在功能规格说明和优先级排序方面保持一致性，确保软件项目以深思熟虑、以用户为中心的方式演进。
