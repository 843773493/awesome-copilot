## Markdown 内容规则

以下 Markdown 内容规则在验证器中被强制执行：

1. **标题**：使用适当的标题级别（H2、H3 等）来组织内容。不要使用 H1 标题，因为这将根据标题自动生成。
2. **列表**：使用项目符号列表或编号列表。确保适当的缩进和间距。
3. **代码块**：使用带围栏的代码块来插入代码片段。指定语言以实现语法高亮。
4. **链接**：使用正确的 Markdown 链接语法。确保链接有效且可访问。
5. **图片**：使用正确的 Markdown 图片语法。为可访问性添加替代文本。
6. **表格**：使用 Markdown 表格来展示表格数据。确保格式和对齐正确。
7. **行长度**：为提高可读性，限制每行长度不超过 400 个字符。
8. **空白符**：使用适当的空白符分隔各部分以提高可读性。
9. **前置内容**：在文件开头包含 YAML 前置元数据，并填写必需的元数据字段。

## 格式和结构

遵循以下指南来格式化和组织您的 Markdown 内容：

- **标题**：使用 `##` 表示 H2，使用 `###` 表示 H3。确保标题以层次结构方式使用。如果内容包含 H4 或更深层次的标题，建议重新组织结构，对于 H5 更加强烈建议。
- **列表**：使用 `-` 表示项目符号列表，使用 `1.` 表示编号列表。嵌套列表请使用两个空格进行缩进。
- **代码块**：使用三个反引号（`）来创建带围栏的代码块。在开头的反引号后指定语言以实现语法高亮（例如，`csharp）。
- **链接**：使用 `[链接文本](URL)` 格式插入链接。确保链接文本具有描述性，且 URL 有效。
- **图片**：使用 `![替代文本](图片URL)` 插入图片。在替代文本中包含图片的简要描述。
- **表格**：使用 `|` 创建表格。确保列正确对齐且包含表头。
- **行长度**：在 80 个字符处换行以提高可读性。使用软换行处理长段落。
- **空白符**：使用空行分隔各部分以提高可读性。避免过多的空白符。

## 验证要求

确保符合以下验证要求：

- **前置内容**：在 YAML 前置元数据中包含以下字段：

  - `post_title`：文章标题。
  - `author1`：文章的主要作者。
  - `post_slug`：文章的 URL 别名。
  - `microsoft_alias`：作者的 Microsoft 别名。
  - `featured_image`：特色图片的 URL。
  - `categories`：文章的分类。这些分类必须来自 /categories.txt 文件中的列表。
  - `tags`：文章的标签。
  - `ai_note`：标明文章是否使用了 AI 生成。
  - `summary`：文章的简要摘要。如有可能，建议根据内容编写摘要。
  - `post_date`：文章的发布日期。

- **内容规则**：确保内容遵循上述 Markdown 内容规则。
- **格式**：确保内容按照指南正确格式化和组织。
- **验证**：运行验证工具以检查是否符合规则和指南。
